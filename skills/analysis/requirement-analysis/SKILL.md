# Requirement Analysis

## 适用场景
- 收到模糊的产品需求，需要拆解为可执行的任务
- 敏捷迭代前的用户故事梳理
- 客户需求澄清会议前的准备
- 评估需求变更的影响范围

## 核心职责
1. 识别需求中的模糊点和隐藏假设
2. 将大需求拆分为独立、可测试的用户故事
3. 为每个用户故事定义明确的验收标准（AC）
4. 识别技术风险和依赖项
5. 估算工作量（T 恤尺码或故事点）

## 分析框架

### 5W2H 检查
- **What**：需求的核心功能是什么？
- **Why**：解决什么痛点？价值主张是什么？
- **Who**：目标用户是谁？角色（Persona）是什么？
- **When**：时间约束？是否有发布窗口？
- **Where**：使用场景？平台（Web/App/小程序）？
- **How**：用户如何操作？核心流程是什么？
- **How much**：数据量？并发量？预算约束？

### 用户故事模板
```
作为 <角色>，
我希望 <目标>，
以便 <价值>。
```

### 验收标准（Given-When-Then）
```
Given <前置条件>
When <用户操作>
Then <预期结果>
```

## 输出格式

### 需求分析报告
```markdown
## 需求概述
- 背景：...
- 目标：...
- 干系人：...

## 澄清问题（Clarifying Questions）
1. ...？（影响范围/优先级）
2. ...？（技术约束）

## 用户故事拆分

### US-1: ...
- **故事**：作为...，我希望...，以便...
- **验收标准**：
  - [ ] AC1: Given... When... Then...
  - [ ] AC2: ...
- **工作量**：M（2-3 天）
- **依赖**：US-X
- **风险**：...

## 技术风险
1. ...

## 建议优先级
P0: US-1, US-2
P1: US-3
```

## 示例

### 示例 1：电商需求分析

**原始需求**：
> "用户想要一个购物车，可以加东西，最后能付款。"

**分析输出**：

#### 澄清问题
1. 购物车是否支持跨设备同步（登录态）？
2. 支持哪些支付方式（微信/支付宝/信用卡）？
3. 商品库存是实时扣减还是下单时扣减？
4. 是否需要支持优惠券和满减活动？

#### 用户故事拆分

**US-1: 添加商品到购物车**
- 故事：作为顾客，我希望将商品加入购物车，以便稍后统一结算。
- AC：
  - Given 商品有库存，When 点击「加入购物车」，Then 购物车数量 +1
  - Given 商品无库存，When 点击「加入购物车」，Then 提示「暂时缺货」

**US-2: 购物车结算**
- 故事：作为顾客，我希望从购物车发起结算，以便完成购买。
- AC：
  - Given 购物车中有商品，When 点击「去结算」，Then 进入订单确认页
  - Given 购物车为空，When 点击「去结算」，Then 提示「购物车是空的」

## 注意事项
- 不要假设用户已知的信息，主动列出需要确认的问题
- 拆分粒度以「一个迭代内可完成」为宜
- 区分「必须做（Must have）」和「锦上添花（Nice to have）」
