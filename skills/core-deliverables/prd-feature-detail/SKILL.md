# PRD-00X 详细需求生成器

## 适用场景
- 已有 PRD-000 概要需求说明书，需要为某个功能模块编写详细需求
- 需要定义具体页面的原型、字段规则、业务逻辑和验收标准
- 需要确保详细需求不推翻概要基线（Scope、NFR、核心实体）

## 核心职责
1. 校验并继承 PRD-000 的基线约束（版本号、Scope、NFR、核心实体）
2. 通过"无情访谈"沿功能树逐分支穷尽模块细节
3. 输出包含用户故事、业务流程、业务逻辑、页面原型、数据字段、接口契约、异常场景的 PRD-00X
4. 执行分层合规检查：确保不引入概要未授权的新实体、不越界
5. 执行质量门禁：模糊语言检测、验收标准可测性检查

## 使用说明

### 如何触发
在 AI 助手对话中输入以下任意语句：
- "概要已确认，帮我写 [模块名] 的详细需求"
- "生成 PRD-001（或 PRD-002、PRD-003...）"
- "基于 PRD-000 v1.0，为 [功能模块] 写详细 PRD"

### 用户配合要点
1. **提供概要文档**：开始前请粘贴或上传对应模块在 PRD-000 中的描述（至少包含模块定位、边界和约束）。
2. **逐功能分支确认**：AI 会按"功能树"逐个分支访谈。每完成一个分支，会生成该部分的草稿。请确认无误后再进入下一个分支。
3. **准备原型草图**（可选）：如果已有手绘原型、Figma 链接或竞品截图，可一并提供，AI 会据此细化交互说明。
4. **验收标准必须可测**：当 AI 输出 AC（验收标准）时，请检查是否包含明确的预期结果和可验证条件。若看到"提升""优化""完善"等模糊词，要求 AI 改写为量化描述。

### 输出物使用方式
- 将生成的文档保存到 `docs\01-requirements\PRD-00X-[模块名].md`（如 `docs\01-requirements\PRD-001-用户与权限管理.md`），若目录不存在则自动创建
- 文档开头的「隶属概要」和「关联详细」用于建立追溯关系
- 所有业务逻辑章节应可直接被开发人员阅读并转化为代码实现

## 执行步骤

### Step 1：继承校验（Baseline Inheritance）
1. 要求用户提供当前模块在 PRD-000 中的定位描述、Scope 边界和 NFR 约束。
2. AI 复述上述约束，向用户确认："以下是我理解的该模块在概要中的边界，是否正确？"
3. 若用户修正，更新理解并再次确认。
4. 明确告知用户："本详细 PRD 不得引入 PRD-000 未定义的新实体，不得修改已冻结的 Scope 和 NFR。"

### Step 2：功能树构建（Feature Tree Construction）
1. 将模块拆解为功能树：
   ```
   模块根节点
   ├── 功能分支 A
   │   ├── 子功能 A1
   │   └── 子功能 A2
   ├── 功能分支 B
   │   ├── 子功能 B1
   │   └── 子功能 B2
   └── 功能分支 C
   ```
2. 向用户展示功能树，确认拆解粒度合适（每个叶子节点对应一个用户故事或一个独立页面）。
3. 按优先级排序功能分支（P0 优先）。

### Step 3：深度访谈（Relentless Interview）

> 访谈策略参考：`references/module-interview-guide.md`

对每个功能分支（从 P0 开始），执行以下子步骤：

1. **用户故事确认**：
   - 使用 `references/acceptance-criteria-patterns.md` 中的模板定义用户故事。
   - 格式：`作为 <角色>，我希望 <目标>，以便 <价值>`。

2. **业务流程梳理**：
   - 绘制该功能分支内的业务流程图（文本形式，如 Mermaid 或步骤列表）。
   - 标注决策节点、异常分支、外部系统调用点。

3. **业务逻辑定义**（重点）：
   - 按 `references/business-logic-spec.md` 定义每个关键处理单元的：
     - **输入**：数据来源、字段、格式、校验规则
     - **处理逻辑**：算法、状态机、业务规则、计算逻辑、权限判断
     - **输出**：返回数据、副作用（数据库更新、消息发送、日志记录）、下游触发
   - 明确列出所有异常分支和对应的处理规则。

4. **页面原型与交互**：
   - 描述关键页面的布局、元素、状态变化。
   - 定义字段列表：字段名、类型、是否必填、校验规则、默认值。

5. **接口契约（内部/外部）**：
   - 定义模块间接口：方法名、入参、出参、错误码。
   - 定义外部 API 调用：接口地址、请求方式、限流、降级策略。

6. **权限与兼容性**：
   - 该功能分支涉及哪些角色？各角色的操作权限如何？
   - 是否需要兼容旧数据/旧接口？

7. **分支确认**：完成一个功能分支后，向用户展示该分支的完整内容，确认无误后再进入下一分支。

### Step 4：分层合规检查（Layer Compliance Check）
全部功能分支访谈完成后，执行以下检查：

1. **实体合规**：详细 PRD 中提到的所有数据实体，是否都能在 PRD-000 的核心实体列表中找到？若发现新实体，标记为"需回溯概要"，不得擅自添加。
2. **Scope 合规**：详细功能是否全部落在 PRD-000 的 In-Scope 内？有无引入 Out-of-Scope 的功能？
3. **NFR 合规**：详细 PRD 中的性能/安全/兼容性指标是否继承自 PRD-000？是否只细化、不推翻？
4. **版本引用**：文档开头是否正确引用了 PRD-000 的版本号？

### Step 5：质量门禁（Quality Gate）
按 `references/acceptance-criteria-patterns.md` 执行：

1. **模糊语言扫描**：全文搜索"优化""完善""提升""友好""快速"等模糊词汇，替换为量化描述。
2. **验收标准可测性**：每个 AC 是否包含 Given-When-Then 结构？预期结果是否可自动化测试或人工核验？
3. **任务分解可行性**：用户故事粒度是否在一个迭代内可完成？过大的故事必须拆分。

### Step 6：输出 PRD-00X
按 `references/feature-template.md` 输出完整文档。

## 示例

### 示例：角色工厂详细需求（片段）

**继承校验**：
> "本模块隶属 PRD-000 v1.0 的'角色工厂'模块。概要约束：角色形象需项目级隔离；生成时间 < 30s。"

**功能树**：
```
角色工厂
├── 角色创建
│   ├── 手动创建
│   └── AI 生成（文生图）
├── 角色管理
│   ├── 编辑属性
│   └── 删除与归档
└── 角色绑定
    └── 绑定到剧本
```

**业务逻辑示例：AI 生成角色形象**：

| 维度 | 说明 |
|------|------|
| **输入** | 角色描述文本（≤500 字）、参考图（JPG/PNG，≤5MB）、风格标签（枚举值） |
| **处理逻辑** | 1. 校验输入文本是否含敏感词<br>2. 调用通义万相 API，传入 prompt + 参考图 URL<br>3. 异步轮询生成状态，超时 30s<br>4. 生成成功后写入素材库，按 project_id 隔离 |
| **输出** | 成功：返回图片 URL、宽高分辨率、生成耗时<br>副作用：写入 `character_images` 表，status=completed<br>下游触发：通知剧本工坊该角色可用 |
| **异常分支** | E1: 敏感词拦截 → 返回具体违规词，保留草稿<br>E2: API 超时 → 保留原图（如有），status=failed，支持重试<br>E3: 图片尺寸超限 → 自动压缩至 2048x2048 |

## 注意事项
- **禁止绕过概要基线**：即使业务上"很想加"某个功能，只要它超出 PRD-000 的 In-Scope 或需要新实体，就必须先回溯修改概要。
- **业务逻辑必须可执行**：写业务逻辑时，想象一个初级开发正在读这段话写代码。如果还有歧义，继续追问用户。
- **参考开源实践**：深度访谈策略参考 Matt Pocock 的 write-a-prd 的"无情访谈"和"深度模块识别"；验收标准模式参考 prd-taskmaster 的 13 项质量检查清单。
