# 验收标准模式与质量门禁

> 参考来源：基于 prd-taskmaster 的 13 项质量检查清单，聚焦验收标准可测性和模糊语言检测。

---

## 验收标准（AC）必须遵循 Given-When-Then

```markdown
Given <前置条件>
When <用户操作或系统事件>
Then <可验证的预期结果>
```

### 合格示例
- ✅ Given 用户已登录且购物车有商品，When 点击"去结算"，Then 跳转订单确认页且商品列表与购物车一致
- ✅ Given 商品库存为 0，When 点击"加入购物车"，Then 按钮置灰并提示"暂时缺货"

### 不合格示例
- ❌ Given 用户登录，When 点击按钮，Then 页面友好地显示结果（"友好"不可测）
- ❌ Given 商品无库存，When 点击购买，Then 系统优化处理（"优化"不可测）

## 模糊语言检测清单

以下词汇出现在 AC 或业务逻辑中时，必须替换为量化描述：

| 模糊词 | 问题 | 替换示例 |
|--------|------|----------|
| 优化 | 无衡量标准 | 响应时间从 3s 降至 1s 以内 |
| 完善 | 无完成标准 | 覆盖所有异常分支（列出具体分支） |
| 提升 | 无基准值 | 转化率从 5% 提升至 8% |
| 友好 | 主观判断 | 操作步骤 ≤ 3 步，有明确错误提示 |
| 快速 | 无量化 | 页面加载 < 2s |
| 大量 | 无规模 | 支持 10 万条数据 |
| 可能/也许 | 不确定性 | 明确是/否，或在何种条件下触发 |
| 等/等等 | 枚举不完整 | 完整列出所有项，或用"包括但不限于：A、B、C" |

## 13 项质量检查清单（精简版）

1. **完整性**：是否覆盖了所有功能分支的正常路径和异常路径？
2. **可测性**：每个 AC 是否能通过测试或演示验证？
3. **无歧义**：是否存在多种理解方式？
4. **一致性**：与 PRD-000 的 Scope、NFR 是否冲突？
5. **可行性**：技术实现难度是否在团队能力范围内？
6. **独立性**：用户故事之间是否耦合过重？
7. **粒度适中**：单个故事是否能在一次迭代内完成？
8. **可追溯**：能否追溯到 PRD-000 的某个概要需求？
9. **无遗漏角色**：是否考虑了所有相关角色的视角？
10. **边界清晰**：输入输出的边界条件是否定义完整？
11. **错误处理**：所有异常场景是否有明确处理规则？
12. **性能暗示**：AC 中是否隐含了性能要求（如"实时"）？
13. **安全暗示**：AC 中是否隐含了权限或数据安全要求？

## 分层合规检查（本体系特有）

在详细 PRD 定稿前，必须额外检查：

1. **实体合规**：详细 PRD 中所有数据实体，是否都能在 PRD-000 的核心实体列表中找到？
2. **Scope 合规**：详细功能是否全部落在 PRD-000 的 In-Scope 内？
3. **NFR 合规**：详细指标是否只细化、不推翻全局 NFR？
4. **版本引用**：文档开头是否正确引用了 PRD-000 版本号？
