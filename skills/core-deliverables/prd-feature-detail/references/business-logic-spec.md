# 业务逻辑编写规范

> 本章规范 PRD-00X 中"业务逻辑"章节的编写标准，确保开发可直接阅读理解并转化为代码。

---

## 业务逻辑的四要素

每个功能点的业务逻辑必须包含以下四个维度：

| 维度 | 说明 | 编写要求 |
|------|------|----------|
| **输入** | 进入本处理单元的数据 | 必须列出：字段名、来源、类型、是否必填、校验规则 |
| **处理逻辑** | 核心算法、规则、状态机 | 必须按步骤描述，避免散文式叙述；涉及判断必须用"如果...则...否则..." |
| **输出** | 离开本处理单元的数据 | 必须列出：返回字段、格式、成功/失败标识 |
| **副作用** | 对外部系统的状态变更 | 必须列出：数据库变更、消息发送、缓存更新、日志、下游触发 |

---

## 输入规范

### 必须描述
- **数据来源**：用户输入 / 上游接口 / 数据库 / 消息队列 / 配置文件
- **字段清单**：字段名、数据类型、长度限制、是否允许为空
- **校验规则**：格式校验（正则）、范围校验（数值范围）、业务校验（唯一性、存在性）
- **示例值**：给出一个有效输入示例

### 输入示例
```markdown
#### 输入
| 字段 | 来源 | 类型 | 必填 | 校验规则 | 示例值 |
|------|------|------|------|----------|--------|
| project_id | 上游接口 | Long | 是 | > 0，且存在 | 10086 |
| character_name | 用户输入 | String | 是 | 1-50 字符，不含特殊符号 | "主角小明" |
| reference_image | 用户上传 | File | 否 | JPG/PNG，≤5MB | avatar.jpg |
```

---

## 处理逻辑规范

### 必须描述
- **主流程**：按编号步骤 1、2、3... 描述
- **判断节点**：用"如果 [条件]，则 [动作]，否则 [替代动作]"
- **状态机**：如果涉及状态流转，必须画出状态迁移表或图
- **算法/规则**：如果有计算逻辑，给出公式或伪代码
- **权限判断**：在什么条件下允许/拒绝执行

### 处理逻辑示例
```markdown
#### 处理逻辑
1. 校验 `project_id` 是否存在且当前用户有写权限
2. 校验 `character_name` 是否在当前项目内唯一
3. 如果上传了 `reference_image`：
   a. 校验图片格式和大小
   b. 上传至 OSS，获取 URL
4. 调用 AI 生成接口，传入 `character_name` + `reference_image_url`（如有）
5. 异步轮询生成状态：
   - 如果 30s 内返回成功：保存图片 URL 和元数据
   - 如果超时：标记状态为 failed，保留原图（如有）
6. 写操作日志：记录用户、时间、输入参数、生成结果
```

## 输出规范

### 必须描述
- **成功输出**：返回数据结构、字段含义、HTTP 状态码（如适用）
- **失败输出**：错误码、错误信息、用户可见提示
- **示例值**：给出成功和失败各一个示例

### 输出示例
```markdown
#### 输出
**成功（200 OK）**
```json
{
  "character_id": 20001,
  "image_url": "https://cdn.example.com/img/20001.png",
  "width": 1024,
  "height": 1024,
  "status": "completed",
  "generated_at": "2026-04-20T14:30:00Z"
}
```

**失败（400 Bad Request）**
```json
{
  "error_code": "CHARACTER_NAME_DUPLICATE",
  "message": "当前项目下已存在同名角色",
  "suggestion": "请更换角色名称"
}
```
```

## 副作用规范

### 必须描述
- **数据库变更**：哪些表被插入/更新/删除？关键字段如何变化？
- **消息通知**：是否发送消息？发送给谁？通过什么渠道？
- **缓存失效**：哪些缓存需要更新或清除？
- **下游触发**：是否会触发其他模块的处理流程？
- **审计日志**：是否记录操作日志？记录哪些字段？

### 副作用示例
```markdown
#### 副作用
- 数据库：`character_images` 表插入一条记录，status = 'completed'
- 缓存：清除 `project:10086:characters` 缓存
- 消息：发送 WebSocket 通知给项目内在线成员："角色 [character_name] 形象已生成"
- 日志：写入 `operation_logs` 表，action = 'CHARACTER_GENERATE'
```

## 异常分支规范

异常分支不是孤立的，必须与主流程的某个步骤对应：

| 异常编码 | 对应主流程步骤 | 触发条件 | 处理规则 | 用户提示 |
|----------|----------------|----------|----------|----------|
| E1 | 步骤 1 | project_id 不存在 | 返回 404，不记录日志 | "项目不存在或已被删除" |
| E2 | 步骤 3a | 图片 > 5MB | 拒绝上传，返回 400 | "图片大小超过 5MB 限制" |
| E3 | 步骤 5 | API 超时 | 状态标记为 failed，保留原图 | "生成超时，请稍后重试" |

## 检查清单

在输出业务逻辑前，自检以下问题：
- [ ] 初级开发阅读后，能否直接开始写代码？
- [ ] 所有输入字段都有类型和校验规则吗？
- [ ] 处理逻辑中，每个"如果"都有对应的"否则"吗？
- [ ] 异常分支覆盖了所有外部依赖失败场景吗？
- [ ] 副作用是否完整到足以让 DBA 和运维理解系统影响？
