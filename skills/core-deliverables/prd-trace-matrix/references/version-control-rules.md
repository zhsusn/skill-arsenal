# PRD 版本联动控制规则

---

## 版本号规则

采用 `主版本.次版本` 格式：

| 版本变更 | 触发条件 | 影响范围 |
|----------|----------|----------|
| 主版本 +1（如 v1.0 → v2.0） | 系统边界、核心实体、Out-of-Scope 发生根本性变更 | 所有关联详细 PRD 必须强制评审 |
| 次版本 +1（如 v1.0 → v1.1） | 补充说明、细化现有内容、修正笔误、NFR 指标调整 | 仅受影响的详细 PRD 需评审 |

---

## 概要升版联动规则

当 PRD-000 发生版本升级时：

1. **自动标记**：所有引用旧版本号的 PRD-00X 自动标记为"待评审"状态。
2. **影响评估**：按 `impact-analysis-guide.md` 执行变更影响分析。
3. **强制评审**：主版本升级时，所有 PRD-00X 必须重新评审；次版本升级时，仅受影响的 PRD-00X 评审。
4. **版本同步**：评审通过后，PRD-00X 的"隶属概要"字段更新为新版本号，PRD-00X 自身次版本 +1。

---

## 详细升版独立规则

当单个 PRD-00X 发生版本升级时：

1. **无需触发概要评审**：若修改仅涉及模块内部细节（字段长度调整、UI 文案优化、新增异常分支），且未触及概要边界。
2. **必须触发概要评审**：若修改涉及以下任一情况：
   - 引入 PRD-000 未定义的新实体
   - 修改或突破全局 NFR
   - 将原 Out-of-Scope 的功能纳入本模块
   - 改变模块间的接口契约（影响其他模块）

---

## 版本引用规范

每份 PRD-00X 开头必须明确引用：

```markdown
> 隶属概要：PRD-000 v{X.X}
```

追溯矩阵中必须记录：

```markdown
| PRD 文档 | 引用概要版本 | 当前概要版本 | 状态 |
|----------|--------------|--------------|------|
| PRD-001 | v1.0 | v1.2 | 待同步 |
```

---

## 状态流转

| 状态 | 含义 | 流转条件 |
|------|------|----------|
| 草稿 | 正在编写 | 初始状态 |
| 已评审 | 通过内部评审 | 完成自检 + 同行评审 |
| 已冻结 | 基线确认，不可随意修改 | 已通过审批流程 |
| 待同步 | 关联概要已升版，需重新评审 | 概要升版后自动标记 |
| 已废弃 | 不再适用 | 功能取消或合并 |

---

## 实践建议

- **版本号写在文件名中**：不要只写 `PRD-001.md`，建议 `PRD-001-用户与权限管理-v1.0.md`
- **变更日志必须说明版本原因**：每次升版，必须在文档控制表中写明"为什么升版"
- **矩阵是唯一的真实来源**：当文档众多时，以 `PRD-REQ-TRACE` 矩阵中的版本记录为准
