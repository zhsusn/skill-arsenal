# PRD 需求追溯矩阵维护器

## 适用场景
- 已有一份 PRD-000 和多份 PRD-00X，需要建立概要 ↔ 详细的血缘关系
- 概要需求变更后，需要快速定位受影响的详细 PRD
- 检查是否有概要功能点没有详细需求承接（悬空需求）
- 生成测试计划骨架（从追溯矩阵推导测试用例覆盖）

## 核心职责
1. 从 PRD-000 和 PRD-00X 中提取需求条目，建立双向追溯关系
2. 生成 REQ-TRACE 矩阵，支持 Markdown 表格和 Excel 格式建议
3. 执行完整性扫描：识别悬空需求、孤儿详细需求、版本引用断裂
4. 执行变更影响分析：给定一个变更点，标红所有关联上下游
5. 维护版本联动规则：概要升版时自动标记关联详细 PRD 待评审

## 使用说明

### 如何触发
在 AI 助手对话中输入以下任意语句：
- "帮我生成需求追溯矩阵"
- "PRD-000 改了，看看哪些详细需求要同步"
- "检查需求有没有遗漏"
- "基于这些 PRD 文件，输出 REQ-TRACE"

### 用户配合要点
1. **提供完整 PRD 集合**：请提供当前最新的 PRD-000 和所有已编写的 PRD-00X 内容（可粘贴或上传文件）。
2. **标注变更点**（变更分析场景）：如果是做变更影响分析，请明确告知"我打算修改 FR-000-05（角色形象管理），会有什么影响？"
3. **确认矩阵格式**：AI 默认输出 Markdown 表格。如果需要导入 Excel，可要求 AI 同时输出 CSV 格式。
4. **定期同步**：每新增或修改一份 PRD-00X，建议重新运行本 skill 更新矩阵。

### 输出物使用方式
- 将追溯矩阵保存到 `docs\01-requirements\PRD-REQ-TRACE-需求追溯矩阵.md`，若目录不存在则自动创建
- 在项目管理工具（Jira/禅道/Notion）中，可将矩阵中的 ID 作为需求单号体系
- 测试团队可直接基于矩阵中的"测试用例"列制定测试计划

## 执行步骤

### Step 1：导入与解析
1. 用户提供 PRD-000 和所有 PRD-00X 的内容。
2. AI 逐文档解析，提取：
   - 概要需求：ID 格式建议为 `FR-000-XX`，描述，所属章节
   - 详细需求：ID 格式建议为 `FR-00X-YY`，描述，隶属 PRD-00X，关联的概要需求 ID
   - 用户故事：US-ID，所属模块，优先级
3. 若发现 PRD-00X 未引用 PRD-000 版本号，或引用的版本号不一致，标记警告。

### Step 2：建立追溯关系
1. 创建矩阵映射：
   ```
   概要需求 ID → 概要描述 → 详细 PRD → 详细需求 ID → 实现状态 → 测试用例 ID
   ```
2. 识别以下异常：
   - **悬空需求（Orphan Outline）**：概要需求 ID 在详细 PRD 中无子需求承接
   - **孤儿详细（Orphan Detail）**：详细需求 ID 找不到对应的概要父需求
   - **版本断裂**：详细 PRD 引用的概要版本号低于当前最新版本

### Step 3：生成 REQ-TRACE 矩阵
按 `references/trace-template.md` 输出完整矩阵。

### Step 4：变更影响分析（按需）

> 分析规则参考：`references/impact-analysis-guide.md`

当用户提出具体变更时：
1. 定位变更涉及的概要需求 ID。
2. 遍历矩阵，找出所有直接关联的详细需求 ID。
3. 进一步找出间接关联（如详细 PRD-A 依赖详细 PRD-B 的接口/数据）。
4. 输出受影响清单，按影响程度分级：
   - 🔴 必须同步修改：详细需求直接继承该概要条目
   - 🟡 建议评审：详细需求间接引用或接口耦合
   - 🟢 无影响：不在该变更链路上

### Step 5：版本联动预警
按 `references/version-control-rules.md`：
1. 若检测到 PRD-000 版本升级，列出所有引用旧版本的 PRD-00X 待评审清单。
2. 建议详细 PRD 的升版策略。

## 输出格式

### REQ-TRACE 矩阵结构

```markdown
# PRD-REQ-TRACE 需求追溯矩阵

> 基于 PRD-000 vX.X 及关联详细 PRD
> 生成日期：YYYY-MM-DD
> 状态：已同步 / 存在异常

## 1. 需求映射表
| 概要需求 ID | 概要描述 | 详细 PRD | 详细需求 ID | 用户故事 ID | 实现状态 | 测试用例 | 备注 |
|-------------|----------|----------|-------------|-------------|----------|----------|------|
| FR-000-05 | 角色形象管理 | PRD-003 | FR-003-01 | US-003-01 | 待开发 | TC-003-01 | |
| FR-000-05 | 角色形象管理 | PRD-003 | FR-003-02 | US-003-02 | 待开发 | TC-003-02 | |
| FR-000-12 | 项目级数据隔离 | PRD-006 | FR-006-03 | US-006-03 | 已验收 | TC-006-03 | |

## 2. 异常报告
### 2.1 悬空需求（无详细承接）
| 概要需求 ID | 概要描述 | 建议动作 |
|-------------|----------|----------|
| FR-000-08 | 批量导入剧本 | 创建 PRD-007 或纳入现有模块 |

### 2.2 孤儿详细（无概要父需求）
| 详细需求 ID | 详细描述 | 所在 PRD | 建议动作 |
|-------------|----------|----------|----------|
| FR-002-99 | 支持 Markdown 编辑器 | PRD-002 | 回溯 PRD-000 补录或删除 |

### 2.3 版本断裂
| PRD 文档 | 引用概要版本 | 当前概要版本 | 建议动作 |
|----------|--------------|--------------|----------|
| PRD-001 | v1.0 | v1.2 | 评审是否受变更影响 |

## 3. 变更影响分析（最近一次）
[按需输出]

## 4. 版本联动待办
[概要升版时输出]
```

## 示例

### 示例：变更影响分析

**用户输入**：
> "PRD-000 v1.1 把'角色形象管理'的存储上限从 100 张改为 500 张，会影响哪些详细需求？"

**AI 输出**：
```markdown
## 变更影响分析：FR-000-05（角色形象管理）

### 🔴 必须同步修改
- **PRD-003 角色工厂**
  - FR-003-01：AI 生成角色形象 → 需更新存储校验逻辑（原 ≤100，现 ≤500）
  - FR-003-02：批量上传参考图 → 需更新批量上限提示

### 🟡 建议评审
- **PRD-002 剧本工坊**
  - FR-002-05：角色绑定 → 绑定列表加载性能（500 张角色时列表渲染是否卡顿）

### 🟢 无直接影响
- PRD-001 用户与权限管理
- PRD-006 系统设置
```

## 注意事项
- **追溯矩阵是活文档**：不是生成一次就结束，每次 PRD 升版都应重新运行本 skill 更新。
- **ID 命名规范**：建议统一前缀，`FR-000-XX`（概要）、`FR-00X-YY`（详细）、`US-00X-YY`（用户故事）、`TC-00X-YY`（测试用例）。
- **间接影响容易被忽略**：变更影响分析不仅要看直接关联，还要看模块间接口和数据依赖。例如修改"角色"的数据结构，可能影响到"剧本绑定角色"的接口契约。
- 本 skill 为完全自建，填补了开源社区在分层需求追溯方面的空白。
