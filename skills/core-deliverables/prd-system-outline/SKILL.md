# PRD-000 概要需求生成器

## 适用场景
- 从零开始规划一个新产品、新系统或大型功能集
- 需要为后续详细需求建立不可推翻的基线约束
- 项目启动前需要明确 Scope、干系人、里程碑和模块边界

## 核心职责
1. 通过三层递进式对话，穷尽收集业务背景、系统边界和成功标准
2. 在每层设置完整性评分（红绿灯机制），不足 90 分则阻塞并追问
3. 输出结构化的 PRD-000 概要需求说明书
4. 定义模块清单，为后续 PRD-00X 详细需求提供输入
5. 明确"冻结基线"：概要中的 Scope、NFR、核心实体在详细阶段不可推翻

## 使用说明

### 如何触发
在 AI 助手对话中输入以下任意语句：
- "我要做一个 [产品/系统名称]，先帮我写概要需求"
- "帮我定一下 [项目名] 的需求基线"
- "生成 PRD-000"

### 用户配合要点
1. **准备材料**（可选但推荐）：提前整理业务背景、竞品分析、现有系统资料，在对话开始时一并提供。
2. **逐层确认**：AI 会将对话分为三层，每层结束后会给出一个"完整性评分"。若标红（<90 分），请根据追问补充信息，不要跳过。
3. **明确说"不"**：当 AI 询问 Out-of-Scope（不在范围内）时，请明确列出"系统不需要做什么"，这对防止后续需求蔓延至关重要。
4. **基线冻结确认**：第三层结束后，AI 会输出完整的 PRD-000 并询问"是否确认基线？"。确认后，该文档中的 Scope、NFR、核心实体定义即被视为冻结。

### 输出物使用方式
- 将生成的 PRD-000 保存到 `docs\01-requirements\PRD-000-概要需求说明书.md`，若目录不存在则自动创建
- 文档末尾的「详细 PRD 清单」即为后续要编写的 PRD-001~PRD-00N 的目录
- 进入详细需求阶段时，使用 `prd-feature-detail` skill，并携带 PRD-000 的版本号

## 执行步骤

### 第一层：问题界定（Problem Framing）
目标：理解"为什么做"和"为谁做"。

1. 向用户索要已有资料（竞品分析、业务文档、数据报表）。
2. 按 `references/questioning-guide.md` 中的「第一层提问集」进行访谈。
3. 输出「问题框架摘要」，包含：业务痛点、目标用户画像、核心价值主张。
4. 按 `references/completeness-scoring.md` 进行第一层评分。
5. 若评分 < 90 分（🔴 红灯或 🟡 黄灯），列出缺失项并追问，直到 ≥ 90 分（🟢 绿灯）。

### 第二层：方案界定（Solution Framing）
目标：确定"系统做什么、不做什么、由哪些模块构成"。

1. 基于第一层成果，激活 `references/system-outline-template.md` 中的模块识别清单。
2. 按 `references/questioning-guide.md` 中的「第二层提问集」进行访谈。
3. 识别系统功能模块，绘制模块边界图（文本形式）。
4. 明确 In-Scope（范围内）与 Out-of-Scope（范围外）。
5. 定义核心实体（如：项目、用户、订单、任务）及其主键/唯一标识。
6. 按 `references/completeness-scoring.md` 进行第二层评分。
7. 若评分 < 90 分，阻塞并追问。

### 第三层：成功标准（Success Criteria）
目标：量化北极星指标、非功能需求和里程碑。

1. 按 `references/questioning-guide.md` 中的「第三层提问集」进行访谈。
2. 定义可量化的北极星指标（如：DAU、转化率、错误率）。
3. 定义全局非功能需求（NFR）：性能、并发、安全、兼容性、可维护性。
4. 制定里程碑与优先级（Phase 1/2/3）。
5. 按 `references/completeness-scoring.md` 进行第三层评分。
6. 若评分 < 90 分，阻塞并追问。

### 输出与冻结
1. 按 `references/system-outline-template.md` 输出完整的 PRD-000。
2. 在文档末尾附加「详细 PRD 清单」，列出每个模块对应的 PRD-00X 编号和责任人。
3. 向用户宣读冻结规则："确认此基线后，PRD-000 中的 Out-of-Scope、NFR、核心实体主键在后续详细 PRD 中不可推翻。如需修改，必须升级 PRD-000 版本号并重新评审所有关联详细 PRD。"
4. 等待用户明确回复"确认"后，标记该文档为 `PRD-000 v1.0 [已冻结]`。

## 输出格式

### PRD-000 概要需求说明书结构

```markdown
# PRD-000 概要需求说明书

> 版本：v1.0
> 状态：已冻结 / 草稿
> 日期：YYYY-MM-DD

## 1. 文档控制
- 版本、作者、审批记录

## 2. 项目背景与目标
- 业务痛点、北极星指标、ROI 预期

## 3. 系统范围（Scope）
### 3.1 In-Scope
### 3.2 Out-of-Scope

## 4. 用户画像与角色矩阵
| 角色 | 职责 | 使用模块 |

## 5. 系统功能架构图
[展示模块及交互关系]

## 6. 核心业务流程（端到端）
[端到端主流程]

## 7. 全局非功能需求（NFR）

## 8. 概要数据模型（核心实体）
仅定义核心实体名称、主键、与其他实体的关系

## 9. 系统集成与约束
- 外部依赖、技术栈约束

## 10. 里程碑与优先级

## 11. 详细 PRD 清单（目录）
| 编号 | 模块名称 | 责任人 | 状态 |
```

## 示例

### 示例：短剧生成平台概要需求

**用户输入**：
> "我们要做一个 AI 短剧生成平台，让用户能从剧本直接生成视频。"

**第一层输出摘要**：
- 痛点：传统短剧制作周期长（2-3 个月），成本高
- 目标用户：独立制片人、MCN 机构内容团队
- 价值主张：剧本到成片时间从月级缩短到天级

**第二层输出摘要**：
- 模块：剧本工坊、角色工厂、分镜工作室、渲染中心、系统设置
- Out-of-Scope：视频分发、版权交易、支付系统

**第三层输出摘要**：
- 北极星指标：单项目从剧本到成片 < 48 小时
- NFR：支持 10 人同时在线编辑；1 万行 Excel 导入 < 5s

**最终 PRD-000 片段**：
```markdown
## 3. 系统范围
### 3.1 In-Scope
- 剧本创作与导入
- AI 角色形象生成
- 分镜自动生成与人工调整
- 视频渲染与下载

### 3.2 Out-of-Scope
- 视频发布到抖音/快手等平台
- 版权交易平台
- 用户充值与支付系统
```

## 注意事项
- **不要一次问完所有问题**。必须分层进行，每层确认完整性后再进入下一层。
- **Out-of-Scope 与 In-Scope 同等重要**。很多需求蔓延源于只定义了"做什么"却没定义"不做什么"。
- **核心实体一旦定义，不可在详细 PRD 中擅自新增**。如果详细阶段发现需要新实体，必须回溯修改 PRD-000 并升版。
- 参考开源实践：本 skill 的递进式对话和红绿灯机制参考了 product-requirement-craft 的三层框架，并适配为 1+N 分层体系。
