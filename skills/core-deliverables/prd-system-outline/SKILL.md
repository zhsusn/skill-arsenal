# PRD-000 概要需求生成器

## 适用场景
- 从零开始规划一个新产品、新系统或大型功能集
- 需要为后续详细需求建立不可推翻的基线约束
- 项目启动前需要明确 Scope、干系人、里程碑和模块边界
- 希望避免 PRD 中出现逻辑矛盾、功能遗漏或技术方案滞后

## 核心职责
1. 通过三层递进式对话，穷尽收集业务背景、系统边界和成功标准
2. 在每层设置完整性评分（红绿灯机制），不足 90 分则阻塞并追问
3. **在输出最终 PRD 前，强制执行内部一致性校验与竞品对标校验**
4. **发现严重矛盾或技术方案缺陷时，向用户列出问题清单并等待确认，禁止自动修正**
5. 输出结构化的 PRD-000 概要需求说明书
6. 定义模块清单，为后续 PRD-00X 详细需求提供输入
7. 明确"冻结基线"：概要中的 Scope、NFR、核心实体在详细阶段不可推翻

## 使用说明

### 如何触发
在 AI 助手对话中输入以下任意语句：
- "我要做一个 [产品/系统名称]，先帮我写概要需求"
- "帮我定一下 [项目名] 的需求基线"
- "生成 PRD-000"

### 用户配合要点
1. **准备材料**（可选但推荐）：提前整理业务背景、竞品分析、现有系统资料，在对话开始时一并提供。
2. **逐层确认**：AI 会将对话分为三层，每层结束后会给出一个"完整性评分"。若标红（<90 分），请根据追问补充信息，不要跳过。
3. **明确说"不"**：当 AI 询问 Out-of-Scope（不在范围内）时，请明确列出"系统不需要做什么"，这对防止后续需求蔓延至关重要。
4. **重视校验阶段**：三层对话结束后，AI 会进入"一致性校验与竞品对标"阶段。若发现严重问题，会列出清单请您确认，**此步骤不可跳过**。
5. **基线冻结确认**：校验通过并输出完整的 PRD-000 后，AI 会询问"是否确认基线？"。确认后，该文档中的 Scope、NFR、核心实体定义即被视为冻结。

### 输出物使用方式
- 将生成的 PRD-000 保存到 `docs\01-requirements\PRD-000-概要需求说明书.md`，若目录不存在则自动创建
- 文档末尾的「详细 PRD 清单」即为后续要编写的 PRD-001~PRD-00N 的目录
- 进入详细需求阶段时，使用 `prd-feature-detail` skill，并携带 PRD-000 的版本号

## 执行步骤

### 第一层：问题界定（Problem Framing）
目标：理解"为什么做"和"为谁做"，收集竞品与技术背景。

1. 向用户索要已有资料（竞品分析、业务文档、数据报表）。
2. 按 `references/questioning-guide.md` 中的「第一层提问集」进行访谈。
3. 若用户未提供竞品信息，AI 应主动基于行业知识列出主流竞品供用户确认或补充。
4. 输出「问题框架摘要」，包含：业务痛点、目标用户画像、核心价值主张、竞品基准。
5. 按 `references/completeness-scoring.md` 进行第一层评分。
6. 若评分 < 90 分（🔴 红灯或 🟡 黄灯），列出缺失项并追问，直到 ≥ 90 分（🟢 绿灯）。

### 第二层：方案界定（Solution Framing）
目标：确定"系统做什么、不做什么、由哪些模块构成"，初步验证技术合理性。

1. 基于第一层成果，激活 `references/system-outline-template.md` 中的模块识别清单。
2. 按 `references/questioning-guide.md` 中的「第二层提问集」进行访谈。
3. 识别系统功能模块，绘制模块边界图（文本形式）。
4. 明确 In-Scope（范围内）与 Out-of-Scope（范围外）。
5. 定义核心实体（如：项目、用户、订单、任务）及其主键/唯一标识。
6. 初步确定技术方案，并与竞品主流方案进行简单对比。
7. 按 `references/completeness-scoring.md` 进行第二层评分。
8. 若评分 < 90 分，阻塞并追问。

### 第三层：成功标准（Success Criteria）
目标：量化北极星指标、非功能需求和里程碑，评估方案可行性。

1. 按 `references/questioning-guide.md` 中的「第三层提问集」进行访谈。
2. 定义可量化的北极星指标（如：DAU、转化率、错误率）。
3. 定义全局非功能需求（NFR）：性能、并发、安全、兼容性、可维护性，附行业对标档位。
4. 制定里程碑与优先级（Phase 1/2/3）。
5. 按 `references/completeness-scoring.md` 进行第三层评分。
6. 若评分 < 90 分，阻塞并追问。

### 第四层：一致性校验与竞品对标（强制校验）
目标：捕获内部矛盾、方案缺陷和技术滞后问题，**未通过本层不得输出最终 PRD**。

1. 激活 `references/consistency-checklist.md`，执行「内部一致性校验」：
   - 检查 In-Scope 与 Out-of-Scope 是否重叠或隐式依赖
   - 检查核心实体是否与功能模块和业务流程匹配
   - 检查 NFR 与技术约束/选型是否兼容
   - 检查角色矩阵与模块权限是否自洽
2. 执行「竞品对标与技术方案校验」：
   - 将当前 In-Scope 功能与竞品核心功能横向对比，识别遗漏
   - 评估当前技术方案是否为主流/最新实践
   - 检查 NFR 是否显著落后于行业标准
   - 检查合规与安全基线是否完整
3. 对发现的问题按 `consistency-checklist.md` 进行分级：
   - 🔴 严重：列出问题清单，使用 checklist 中的「用户确认模板」向用户展示，**等待用户明确答复**
   - 🟡 建议：列出为「待决策项」，写入 PRD 第 12 章
   - 🟢 提示：直接写入 PRD「风险提示」章节
4. **若存在未解决的 🔴 严重问题，禁止进入"输出与冻结"阶段**。
5. 校验完成后，向用户公示「校验摘要」。

### 输出与冻结
1. 按 `references/system-outline-template.md` 输出完整的 PRD-000。
2. 在文档末尾附加「详细 PRD 清单」，列出每个模块对应的 PRD-00X 编号和责任人。
3. 向用户宣读冻结规则："确认此基线后，PRD-000 中的 Out-of-Scope、NFR、核心实体主键在后续详细 PRD 中不可推翻。如需修改，必须升级 PRD-000 版本号并重新评审所有关联详细 PRD。"
4. 等待用户明确回复"确认"后，标记该文档为 `PRD-000 v1.0 [已冻结]`。

## 输出格式

### PRD-000 概要需求说明书结构

```markdown
# PRD-000 概要需求说明书

> 版本：v1.0
> 状态：已冻结 / 草稿
> 日期：YYYY-MM-DD

## 1. 文档控制
- 版本、作者、审批记录

## 2. 项目背景与目标
- 业务痛点、北极星指标、ROI 预期

## 3. 竞品与差异化分析
- 直接竞品对标表、间接替代方案、技术方案对比

## 4. 系统范围（Scope）
### 4.1 In-Scope
### 4.2 Out-of-Scope

## 5. 用户画像与角色矩阵
| 角色 | 职责 | 使用模块

## 6. 系统功能架构图
[展示模块及交互关系]

## 7. 核心业务流程（端到端）
[端到端主流程]

## 8. 全局非功能需求（NFR）
（附行业对标档位）

## 9. 概要数据模型（核心实体）
仅定义核心实体名称、主键、与其他实体的关系

## 10. 系统集成与约束
- 外部依赖、技术栈约束

## 11. 里程碑与优先级

## 12. 风险提示与待决策项
- 已确认风险、待决策项、校验摘要

## 13. 详细 PRD 清单（目录）
| 编号 | 模块名称 | 责任人 | 状态 |
```

## 示例

### 示例：短剧生成平台概要需求

**用户输入**：
> "我们要做一个 AI 短剧生成平台，让用户能从剧本直接生成视频。"

**第一层输出摘要**：
- 痛点：传统短剧制作周期长（2-3 个月），成本高
- 目标用户：独立制片人、MCN 机构内容团队
- 价值主张：剧本到成片时间从月级缩短到天级
- 竞品基准：Runway（AI 视频生成）、剪映（剪辑工具）

**第二层输出摘要**：
- 模块：剧本工坊、角色工厂、分镜工作室、渲染中心、系统设置
- Out-of-Scope：视频分发、版权交易、支付系统
- 技术约束：云端 GPU 渲染，支持主流视频格式导出

**第三层输出摘要**：
- 北极星指标：单项目从剧本到成片 < 48 小时
- NFR：支持 10 人同时在线编辑；1 万行 Excel 导入 < 5s
- 行业对标：渲染性能处于中档，低于 Runway 但高于一般 SaaS

**第四层（校验示例）**：
- 🔴 严重问题发现：Out-of-Scope 包含"支付系统"，但 In-Scope 未明确"用户如何付费使用高级功能"，商业模式存在逻辑缺口。
- 向用户确认后：用户补充"第一阶段采用免费+申请试用，支付系统放入 Phase 2"，矛盾解除。

**最终 PRD-000 片段**：
```markdown
## 4. 系统范围
### 4.1 In-Scope
- 剧本创作与导入
- AI 角色形象生成
- 分镜自动生成与人工调整
- 视频渲染与下载

### 4.2 Out-of-Scope
- 视频发布到抖音/快手等平台
- 版权交易平台
- 用户充值与支付系统（Phase 1 不做，放入 Phase 2 规划）

## 12. 风险提示与待决策项
### 12.1 已确认风险
- Phase 1 无支付系统，高级功能采用免费+申请试用模式，存在羊毛党风险 → 用户确认接受

### 12.3 校验摘要
- 内部一致性校验：通过（修正 1 处商业模式逻辑缺口）
- 竞品对标校验：通过（发现 1 项待决策项：是否引入 AI 语音克隆，对标 ElevenLabs）
```

## 注意事项
- **不要一次问完所有问题**。必须分层进行，每层确认完整性后再进入下一层。
- **第四层校验不可跳过**。即使前三层评分都是绿灯，也必须在输出前执行一致性校验和竞品对标。
- **发现严重问题时禁止自动修改**。必须列出问题清单，等待用户确认后再修正 PRD。
- **Out-of-Scope 与 In-Scope 同等重要**。很多需求蔓延源于只定义了"做什么"却没定义"不做什么"。
- **核心实体一旦定义，不可在详细 PRD 中擅自新增**。如果详细阶段发现需要新实体，必须回溯修改 PRD-000 并升版。
- 参考开源实践：本 skill 的递进式对话和红绿灯机制参考了 product-requirement-craft 的三层框架，并适配为 1+N 分层体系。
