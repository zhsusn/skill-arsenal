# PRD-000 一致性校验与竞品对标清单

> 本清单在生成最终 PRD-000 前强制执行，用于捕获内部矛盾、方案缺陷和技术滞后问题。

---

## 校验流程总览

生成最终 PRD-000 前，必须执行以下两步校验：

1. **内部一致性校验**：检查 PRD 各章节之间是否存在逻辑矛盾。
2. **竞品对标与技术方案校验**：将当前方案与行业主流竞品/技术进行横向对比。

校验结果按严重度分级，**严重问题必须在输出最终 PRD 前与用户确认并解决**，建议性问题可标注为"待决策项"。

---

## 第一步：内部一致性校验

逐条检查，发现矛盾时记录问题编号、位置和修正建议。

### 1.1 Scope 自洽性
- [ ] **In-Scope 与 Out-of-Scope 重叠检查**：是否存在同一功能既在 In-Scope 又在 Out-of-Scope 中？
- [ ] **Out-of-Scope 边界合理性**：Out-of-Scope 中的功能是否被业务流程或角色矩阵隐式依赖？（例如：Out-of-Scope 写了"不做支付"，但 In-Scope 包含"会员购买"）
- [ ] **范围与里程碑匹配**：Phase 1 的交付物是否超出了最小可用闭环？是否存在 Phase 1 依赖了 Out-of-Scope 的功能？

### 1.2 实体与模块一致性
- [ ] **实体覆盖度**：每个核心实体是否至少被一个 In-Scope 模块使用？是否存在"僵尸实体"（定义了但无模块维护）？
- [ ] **实体关系与流程一致性**：核心业务流程中涉及的实体操作（创建/读取/更新/删除）是否与实体关系定义冲突？
- [ ] **主键稳定性**：各实体主键是否满足全局唯一性？是否存在不同模块对同一实体使用不同标识的风险？

### 1.3 NFR 与功能需求一致性
- [ ] **性能目标可行性**：性能指标（如并发、响应时间）是否与系统架构和技术约束兼容？（例如：单机部署却要求百万 QPS）
- [ ] **安全与 Scope 一致性**：安全等级要求是否与 Out-of-Scope 的安全措施冲突？（例如：要求等保三级但 Out-of-Scope 包含"身份认证"）
- [ ] **兼容性与技术栈一致性**：兼容性要求（如 IE11 支持）是否与既定技术栈矛盾？

### 1.4 角色与权限一致性
- [ ] **角色覆盖度**：每个模块是否至少被一个角色使用？是否存在无角色的模块？
- [ ] **权限闭环**：角色的职责描述是否与其使用模块的权限等级匹配？是否存在"低权限角色操作高风险功能"的冲突？

### 1.5 依赖与约束一致性
- [ ] **外部依赖闭环**：外部依赖系统（如支付、短信）是否被 In-Scope 功能调用，但列在 Out-of-Scope 或未明确接口？
- [ ] **技术约束冲突**：既定技术栈中是否包含互相排斥的技术？（例如：前端框架同时选用了 React 和 Vue）

---

## 第二步：竞品对标与技术方案校验

基于用户提供（或 AI 主动检索）的竞品信息，进行横向对比。

### 2.1 功能完整性对标
- [ ] **核心功能遗漏**：竞品已有的、且与本系统定位直接相关的核心功能，当前 In-Scope 是否遗漏？
- [ ] **差异化定位**：当前方案与竞品相比，差异化优势是否明确？是否存在"做了一样的事情但毫无优势"的风险？
- [ ] **边缘场景覆盖**：竞品已解决的常见边缘场景（如异常状态、批量操作、撤回机制），当前方案是否考虑？

### 2.2 技术方案先进性校验
- [ ] **架构模式时效性**：当前采用的架构模式是否为领域主流？是否存在明显过时的方案？（例如：单体架构做超大规模分布式系统）
- [ ] **技术栈版本**：推荐的技术栈是否为当前 LTS/稳定版本？是否存在已停止维护的依赖？
- [ ] **性能方案合理性**：满足 NFR 的性能指标，所选技术方案是否为当前最优实践？是否有更轻量、成本更低的替代方案未考虑？
- [ ] **AI/自动化替代**：当前人工流程是否可通过 AI 或自动化工具优化？竞品是否已采用相关技术？

### 2.3 行业标准与合规
- [ ] **合规基线**：同行业的合规要求（如金融行业的等保、医疗行业的 HIPAA、出海产品的 GDPR）是否完整覆盖？
- [ ] **安全基线**：同类型产品的常见安全机制（如 MFA、审计日志、数据脱敏）是否已纳入 In-Scope？

## 问题分级与处理规则

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| 🔴 **严重（阻塞）** | 会导致 PRD 无法执行、存在根本逻辑矛盾或重大功能遗漏 | **必须**向用户提出，获得明确答复后才能输出最终 PRD |
| 🟡 **建议（待决策）** | 方案可执行但存在优化空间，或竞品有更好的实践 | 列出为「待决策项」，由用户决定是否采纳 |
| 🟢 **提示（信息补充）** | 不影响执行，但补充后可使 PRD 更完整 | 直接写入 PRD「风险提示」章节，无需阻塞 |

### 用户确认模板

发现 🔴 严重问题时，使用以下模板向用户展示，禁止自动修改 PRD：

```
## ⚠️ 校验发现严重问题，需您确认

在生成最终 PRD-000 前，我发现以下 [N] 个严重问题，可能影响方案可行性：

### 问题 1：[一句话概括]
- **位置**：第 X 章 - [具体章节名]
- **矛盾点**：[具体问题描述]
- **影响**：[如果不修复会怎样]
- **建议方案**：[AI 推荐的修正方向]
- **请您确认**：A. 按建议调整  B. 保持原方案并说明原因  C. 提出其他方案

### 问题 2：...

待您确认后，我将生成修正后的 PRD-000。
```

## 校验输出物

校验完成后，生成以下两份内容并附在最终 PRD 之前：

1. **《校验摘要》**（向用户展示）：
   - 校验通过项数 / 发现问题数
   - 严重问题清单（🔴，需用户确认）
   - 待决策项清单（🟡，可选采纳）
   - 提示项清单（🟢，已自动记录）

2. **《PRD-000 风险提示》**（写入 PRD 第 12 章）：
   - 已解决的矛盾
   - 用户确认保留的已知风险
   - 待决策项及建议决策时间点
