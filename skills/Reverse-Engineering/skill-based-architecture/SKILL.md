# Skill-Based Architecture

## 适用场景

- 项目 AI 文档超过 150 行且存在重复
- 需要在多个 AI 工具间共享项目规则（Cursor、Claude、Codex、Windsurf、Gemini 等）
- 希望减少每次任务的 token 浪费
- 需要自动捕获和沉淀项目经验教训
- 正在从零搭建项目级 AI skill 体系

## 目标结构

```
skills/<name>/
├── SKILL.md          # <= 100 行：Always Read + 任务路由表
├── rules/            # 长期约束（始终为真）
├── workflows/        # 分步流程（如何做事）
├── references/       # 背景知识：架构、陷阱、索引
│   └── gotchas.md    # 已知陷阱——往往价值最高
└── docs/             # 可选：提示词、报告、外部文档
```

所有根入口文件（`AGENTS.md`、`CLAUDE.md`、`CODEX.md`、`.cursor/rules/*.mdc`、`.windsurf/rules/*.md`）均变为**薄壳**（<= 15 行），仅包含路由表和指向正式 skill 的指针。

## 核心原则

### 1. SKILL.md 是路由器

- 导航到正确文件，绝不穷尽；**<= 100 行**
- **Always Read**（2-3 个文件，~150 行）—— 每个任务都加载
- **Common Tasks**（任务路由）—— 仅加载当前任务所需的文件

### 2. 薄壳 + 内联路由表

- 所有入口文件不超过 15 行
- 嵌入**内联路由表**，经受上下文压缩后仍可被识别
- 自然语言指令在长对话中会被压缩丢失，路由表不会

### 3. Description = 触发条件

- 描述字段不是被动摘要，而是明确的激活条件
- 必须包含 >= 2 个带引号的触发短语
- 坏：`description: 帮助整理项目规则`
- 好：`description: 当用户要求"整理项目规则"或"重构为 skill-based architecture"时激活`

### 4. 内容分类

| 内容类型 | 存放位置 | 示例 |
|---------|---------|------|
| 稳定约束 | `rules/` | 命名规范、模块边界、依赖策略 |
| 分步流程 | `workflows/` | add-controller、fix-bug、release |
| 架构、陷阱、索引 | `references/` | 系统设计、gotchas、路由表 |
| 边界情况、踩坑 | `references/gotchas.md` | 生命周期陷阱、时序依赖 |
| 提示词、报告 | `docs/` | 模板、生成报告 |
| 编辑器配置 | `.cursor/`、`.claude/` | 仅限薄壳，不含规则内容 |

### 5. Session Discipline

- 同一会话中的**每个新任务**都必须重新读取 SKILL.md 并重新匹配路由
- "我之前读过了" 不是有效理由——上下文会静默压缩；新任务可能匹配不同路由；部分记忆比没有记忆更糟

### 6. 任务闭环协议（After-Action Review）

- 每个非平凡任务完成前执行 30 秒复盘
- **4 个问题**：新 pattern？新陷阱？缺失规则？过时规则？
- **记录阈值**：满足 2/3 才记录（可重复 + 代价高 + 不明显）
  - **可重复**：未来任务是否会再次遇到？
  - **代价高**：忽略它是否会浪费 30+ 分钟调试？
  - **不明显**：未来读者能否仅从代码中推断出来？

### 7. 激活优先于存储

- 记录在 `references/` 中的陷阱如果不在任务路径中，未来 agent 可能永远不会读到
- 高价值教训必须在相关 `workflows/*.md` 的完成检查清单或 SKILL.md 路由中显式引用
- 如果教训是稳定约束，应提升为 `rules/`

### 8. 通用化规则

- 记录必须是**可复用的知识**，而非项目特定的叙事
- 坏：`在产品模块中，切换标签页时需要重置分页`
- 好：`切换上下文（标签页、视图、筛选器）时，将分页重置为第 1 页`

### 9. 自我维护

- **文件健康检查**：大小扫描（参考范围而非硬限制）
- **评估式拆分**：仅在主题真正可分离时才拆分
- **碎片合并**：将属于一起的微小文件合并（50-200 行/文件为佳）
- **规则废弃**：显式流程移除过时规则
- **引用完整性**：重命名、拆分、删除后检查链接

## 迁移路径

| 条件 | 路径 |
|------|------|
| 总规则 < 150 行，无重复，无反复踩坑 | **最小化单 SKILL.md** — 使用入门模板 |
| 规则 > 150 行但无重复 | **Quick Start 脚手架** — 运行脚本，填充 TODO |
| 规则 > 150 行 **且** 跨文件重复 | **完整 9 阶段迁移** |

### 完整迁移（9 阶段）

| 阶段 | 内容 |
|------|------|
| **1. Audit** | 盘点所有规则来源：SKILL.md、AGENTS.md、CLAUDE.md、.cursor/rules/、README、docs/ |
| **2. Design** | 基于项目规模规划文件集（最小化 / 典型 / 领域特定） |
| **3. Write SKILL.md** | 创建 <= 100 行的入口：Always Read + Common Tasks + Known Gotchas |
| **4. Extract Rules** | 将稳定约束移入 `rules/`（project-rules、coding-standards、domain rules） |
| **5. Extract Workflows** | 创建工作流 + 元工作流（update-rules、maintain-docs） |
| **6. Extract References** | 将架构概述、gotchas、源索引移入 `references/` |
| **7. Create Entry Points** | 为所有工具创建薄壳入口 + 内联路由表 |
| **8. Verify** | 运行自动化 `smoke-test.sh`（48 项结构、路由、占位符、激活检查） |
| **9. Pressure-Test** | 在时间/沉没成本/权威压力下派遣 subagent；将理性化表述录入合理化表 |

支持增量迁移——按轮次进行，不阻塞日常工作。

## 常见陷阱

| 陷阱 | 影响 | 修复 |
|------|------|------|
| **缺少 Cursor 注册入口** | Cursor 永远无法发现 skill；所有规则被静默忽略 | 创建 `.cursor/skills/<name>/SKILL.md` |
| **仅软指针的薄壳** | 上下文压缩后指令丢失 | 在每个入口文件中嵌入内联路由表 |
| **描述模糊** | Skill 存在但 agent 从不激活 | 写 >= 20 词，包含 >= 2 个带引号的触发短语 |
| **存了但未激活** | 陷阱在 `references/` 中但不在任何工作流里 | 同时在工作流检查清单或 SKILL.md 路由中显式引用 |
| **跳过任务闭环** | 经验教训未被捕获；重复犯相同错误 | 将任务闭环协议作为完成门禁 |
| **同一会话中跳过路由** | agent 读取 task 1 的 SKILL.md 后跳过 task 2 | Session Discipline 规则 + 所有薄壳中的重读触发器 |
| **项目特定记录** | 超出当前上下文后无用 | 记录前应用通用化规则 |

## 反模式

| 反模式 | 危害 | 修复 |
|--------|------|------|
| 肥薄壳（50+ 行） | 两个地方更新；破坏单一真相源 | 精简到 <= 15 行 |
| SKILL.md 当第二个 README | 冗余上下文；超过 100 行 | 安装指南放 README；SKILL.md 只负责导航 |
| 规则和流程混合 | 难以查找检查清单；难以更新约束 | 约束放 `rules/`，流程放 `workflows/` |
| 巨型子文件（500+ 行） | 和过大的 SKILL.md 一样的问题，只是下移一层 | 按子域拆分 |
| 过度拆分（20 个文件，每份 10 行） | 导航开销超过收益 | 合并相关文件；目标 50-200 行/文件 |
| 什么都记录 | 规则被低价值噪音膨胀 | 应用记录阈值：满足 2/3 标准 |

## 何时不使用

- **短期个人项目**（< 2 周）—— 无反复任务，无值得捕获的规则
- **总规则内容 < 50 行** —— 单文件 `CLAUDE.md` 或 `.cursorrules` 足够
- **单一工具** —— 你只用一种 AI 工具且不需要跨工具兼容
- **无团队共享** —— 你是唯一在代码库上使用 AI agent 的人，且小到可以记在脑中

在这些情况下，从普通的 `CLAUDE.md` 或 `.cursor/rules/workflow.mdc` 开始即可。项目成长后随时可以迁移。

## 快速开始

### 脚手架新项目

1. 确定项目名 `NAME` 和一句话摘要 `SUMMARY`
2. 复制模板树到 `skills/$NAME/`
3. 替换所有 `{{NAME}}` 和 `{{SUMMARY}}` 占位符
4. 填充每个 `<!-- FILL: -->` 标记（未解决会导致 skill 静默激活失败）
5. 运行 `smoke-test.sh` 验证 48 项检查
