# 技术设计文档生成器

根据已冻结的产品需求文档（PRD），生成结构化的软件设计文档（SDD）。采用八章节体系，确保设计决策可追踪、可评审、可落地。

## 适用场景

- 基于已审批 PRD 编写技术设计文档
- 系统架构设计与文档化
- API 接口设计与契约定义
- 数据库 Schema 与数据流设计
- 技术方案评审与选型论证

## 文档结构概览

| 章节 | 内容 |
|------|------|
| 1. 引言 | 目的、范围、术语定义、参考资料 |
| 2. 系统概述 | 高层架构、系统上下文、关键组件 |
| 3. 设计考量 | 假设、约束、依赖、风险 |
| 4. 架构策略 | 关键决策、备选方案对比 |
| 5. 系统架构 | 组件图、数据流、API、数据模型 |
| 6. 策略与机制 | 安全、错误处理、日志监控、性能 |
| 7. 详细设计 | 组件级规格说明 |
| 8. 附录 | 时序图、状态图、术语表 |

## 章节模板

### 第 1 章：引言

```markdown
## 1. 引言

### 1.1 目的
[本文档覆盖范围及目标读者]

### 1.2 范围
[系统边界——包含与不包含的内容]

### 1.3 术语与缩写
| 术语 | 定义 |
|------|------|
| [术语] | [定义] |

### 1.4 参考资料
- PRD: [链接]
- [其他相关文档]
```

### 第 2 章：系统概述

```markdown
## 2. 系统概述

### 2.1 系统上下文
[本系统在整体架构中的位置]

### 2.2 高层架构
[架构图描述或 ASCII 图]

### 2.3 关键组件
- **组件 A**: [职责]
- **组件 B**: [职责]
```

### 第 3 章：设计考量

```markdown
## 3. 设计考量

### 3.1 假设
- [假设 1]
- [假设 2]

### 3.2 约束
- [技术约束]
- [业务约束]

### 3.3 依赖
- [外部系统 X]
- [库 Y 版本 Z]

### 3.4 风险
| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| [风险] | 高/中/低 | [策略] |
```

### 第 4 章：架构策略

```markdown
## 4. 架构策略

### 4.1 选型理由
[为何选择当前架构而非备选]

### 4.2 备选方案
| 方案 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| [方案 A] | [优点] | [缺点] | 选中 |
| [方案 B] | [优点] | [缺点] | 放弃 |

### 4.3 关键架构决策
- **决策 1**: [决策内容及理由]
- **决策 2**: [决策内容及理由]
```

### 第 5 章：系统架构

```markdown
## 5. 系统架构

### 5.1 组件图
[描述或 ASCII 表示]

### 5.2 数据流
[数据如何在系统中流转]

### 5.3 API 设计

#### 5.3.1 端点
| 方法 | 路径 | 描述 |
|------|------|------|
| POST | /api/resource | 创建资源 |
| GET | /api/resource/:id | 获取资源 |

#### 5.3.2 请求/响应模式
[TypeScript 接口或 JSON Schema]

### 5.4 数据模型
[数据库 Schema，含类型与约束]
```

### 第 6 章：策略与机制

```markdown
## 6. 策略与机制

### 6.1 安全
- 认证: [策略]
- 授权: [策略]
- 数据加密: [静态 / 传输中]

### 6.2 错误处理
- [错误处理策略]
- [重试策略]

### 6.3 日志与监控
- [记录内容]
- [告警阈值]

### 6.4 性能
- [缓存策略]
- [连接池]
```

### 第 7 章：详细设计

```markdown
## 7. 详细设计

### 7.1 [组件 A] 设计

#### 7.1.1 职责
- [职责 1]
- [职责 2]

#### 7.1.2 接口
[TypeScript 接口或函数签名]

#### 7.1.3 实现说明
[具体实现细节、算法或模式]
```

### 第 8 章：附录

```markdown
## 8. 附录

### 8.1 时序图
[复杂流程]

### 8.2 状态图
[有状态组件]

### 8.3 术语表
[扩展定义]
```

## 需求可追溯性

每项设计决策应能追溯到需求：

```markdown
### REQ-001 实现
**需求:** [引用需求原文]
**设计:** [本设计如何满足该需求]
**验证:** [如何测试]
```

## 质量检查清单

文档定稿前确认：

- [ ] 每个 PRD 需求（REQ-XXX）均已回应
- [ ] 数据模型完整，含类型与约束
- [ ] API 契约已完整定义
- [ ] 错误场景已文档化
- [ ] 安全考量已覆盖
- [ ] 性能需求有对应策略
- [ ] 依赖项已标注版本
- [ ] 在有帮助的地方包含图表
