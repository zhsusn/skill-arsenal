# AI 架构顾问

为基于大语言模型的项目提供架构指导。通过分析需求复杂度、复用性、上下文管理和安全约束，推荐 Skill、Agent、直接提示词与 SDK 原语的最佳组合。

## 适用场景

- 设计 AI 驱动的项目架构
- 选择合适的 AI 交互原语
- 优化上下文管理与成本控制
- 规划多 Agent 协作工作流
- 评估安全边界与工具权限

## 核心能力

- **需求分析**：理解项目复杂度、范围与约束
- **架构推荐**：Skill vs Agent vs 提示词 vs SDK 原语
- **决策评估**：基于评分框架的量化选型
- **渐进式披露**：高效的上下文管理设计
- **安全考量**：构建安全、可控的 AI 系统

## 四大架构原语

### 1. Skill（可复用领域知识库）

**定义**：由指令、脚本和资源组成的有组织文件夹，供 AI 动态发现与加载。

**何时使用**：
- ✅ 需要专业化领域知识
- ✅ 跨多个项目复用
- ✅ 复杂的多步骤工作流
- ✅ 需要参考资料
- ✅ 渐进式披露有益

**何时不用**：
- ❌ 简单、一次性任务
- ❌ 仅项目专属逻辑
- ❌ 无需复用

### 2. Agent / Subagent（自主任务处理器）

**定义**：具备独立系统提示词、专属上下文窗口和特定工具权限的专用代理。

**何时使用**：
- ✅ 复杂的多步骤自主任务
- ✅ 需要隔离上下文
- ✅ 不同工具权限需求
- ✅ 并行任务执行
- ✅ 按任务类型分配专家

**何时不用**：
- ❌ 简单查询或查找
- ❌ 需要共享上下文
- ❌ 顺序强依赖
- ❌ 资源受限环境

### 3. 直接提示词（简单指令）

**定义**：直接传递给 AI 的清晰、明确的指令，无额外结构。

**何时使用**：
- ✅ 简单、直接的任务
- ✅ 一次性操作
- ✅ 快速问答
- ✅ 无需专业化
- ✅ 上下文需求极小

**何时不用**：
- ❌ 复杂多步骤流程
- ❌ 需要复用
- ❌ 需要领域专业知识

### 4. SDK 原语（自定义工作流）

**定义**：使用 AI Agent SDK 的低级构建块创建自定义代理工作流。

**何时使用**：
- ✅ 独特工作流需求
- ✅ 自定义工具集成
- ✅ 特定反馈循环
- ✅ 与现有系统集成
- ✅ 需要细粒度控制

**何时不用**：
- ❌ 标准场景已被 Skill/Agent 覆盖
- ❌ 开发资源有限
- ❌ 快速上线优先

## 快速决策矩阵

| 评估维度 | 提示词 | Skill | Agent | SDK |
|---------|--------|-------|-------|-----|
| **任务复杂度** | 低（1-3） | 中（4-6） | 高（7-9） | 定制（10） |
| **复用性** | 单次 | 团队级 | 组织级 | 产品特性 |
| **上下文管理** | 极简 | 结构化 | 隔离 | 定制 |
| **安全要求** | 基础 | 受控 | 高 | 最高 |
| **性能需求** | 标准 | 快速响应 | 并行执行 | 极致优化 |
| **维护成本** | 无 | 中 | 高 | 定制 |
| **上市时间** | 即时 | 数小时 | 数日 | 数周+ |

**评分方法**：为上述 7 个维度分别打分（1-10），按加权平均计算总分。
- **1-3 分**：直接提示词
- **4-6 分**：Skill
- **7-9 分**：Agent / Subagent
- **10 分**：SDK 原语

默认权重：复杂度 25%、复用性 20%、上下文 15%、安全 15%、性能 10%、维护 10%、上市时间 5%。

## 架构模式

### 模式一：Skill 优先架构

**适用**：构建可复用的专业知识与流程

```
Project
├── skills/
│   ├── domain-expert/
│   │   ├── SKILL.md
│   │   └── references/
│   └── workflow-automation/
│       ├── SKILL.md
│       └── scripts/
└── .cursor/
    └── config
```

**优势**：跨项目复用、渐进式披露、易于分享维护、文档清晰。

### 模式二：Agent 流水线架构

**适用**：复杂自主任务，各阶段需隔离

```
主 Agent（编排器）
├── 探索 Agent（代码库分析）
├── 规划 Agent（任务规划）
├── 编码 Agent（实现）
└── 审查 Agent（验证）
```

**优势**：并行执行、上下文隔离、专业化分工、职责清晰。

### 模式三：混合架构

**适用**：复杂项目，需求多样化

```
主对话
├── 直接提示词（简单任务）
├── Skill（可复用专业知识）
│   ├── code-review-skill
│   └── testing-skill
└── Subagent（复杂工作流）
    ├── 探索 Agent
    └── 规划 Agent
```

**优势**：各任务使用合适工具、资源最优、灵活可扩展。

### 模式四：SDK 自定义架构

**适用**：独特需求或产品化特性

```
自定义 Agent SDK 实现
├── 自定义工具
├── 专用反馈循环
├── 系统集成
└── 领域特定工作流
```

**优势**：完全可控、自定义集成、独特工作流、生产就绪。

## 核心设计原则

1. **渐进式披露**：仅在需要时展示所需信息。通过任务导航、查询工具、层级结构优化上下文使用。
2. **上下文即资源**：将上下文窗口视为珍贵且有限的资源。优先检索而非倾倒，定期压缩与重置。
3. **清晰明确**：给出具体、无歧义的指令。定义输出格式、提供示例、避免模糊表达。
4. **安全设计**：默认拒绝，白名单方式。限制工具访问、敏感操作需确认、审计所有操作。
5. **思考能力**：对复杂推理启用深度思考模式。要求逐步思考、工具使用后允许反思。
6. **双消息模式**：使用元消息提供上下文而不干扰 UI。适用于 Skill 加载等场景。

## 常见反模式

| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| 过度工程 | 对简单任务使用 Agent/SDK | 从简单开始，按需扩展复杂度 |
| 上下文倾倒 | 一次性加载整个 Skill | 使用渐进式披露、查询工具 |
| 混合关注点 | 单个 Skill/Agent 做太多事 | 分离关注点，使用 Subagent 或多 Skill |
| 无安全边界 | 所有 Agent 拥有完整工具权限 | 白名单方式，最小权限原则 |
| 忽视复用性 | 重复创建相同提示词 | 提取为 Skill，跨项目共享 |
